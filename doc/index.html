@verificatum/bundle

Self-contained cryptographic library for use in electronic voting clients. Complete documentation with explicit references to the literature on basic arithmetic algorithms makes it good teaching material.

This is still a development repo that may break now and then :-)

Installation

Installation instructions are given for Ubuntu. Adapt as needed.

# Building tools.
sudo apt install make m4

# Most recent version of node.
curl -fsSL https://deb.nodesource.com/setup_current.x | sudo -E bash -
sudo apt install -y nodejs

# TypeScript globally.
sudo npm install -g typescript

# Install NPM development dependencies.
npm i

# To enable WASM code the following is needed.
sudo apt install clang
sudo update-alternatives --install /usr/bin/wasm-ld wasm-ld /usr/bin/wasm-ld-12 100

# To run the webserver that serves the test and benchmark suites you
# may need to install Python 3.
sudo apt install python3

Usage

The simplest way to use the library is to use one of the single-file JavaScript files compiled using make jsfiles which end up in pure/dist, but users may also choose to use a bundling tool that understands monorepos.

Only here for the fast integer arithmetic? Build using make vts-ba and include pure/dist/vts-ba-VERSION.js in your project.

Examples

After installing the build tools above you can build the library and start a webserver at localhost where you can try it directly. Simply use make webserver to compile the code and start the webserver. The examples, benchmark suite, and test suite are then available on your computer at http://localhost:8000.

Documentation

The TypeDoc documentation can be built using make doc and end up in the subdirectory doc, but it is also available here. We have modified the output to link directly to the relevant lines in the actual source code at GitHub.

We have made an effort to not only cite sources for any non-trivial algorithms, but also adhere to the notation in the pseudo code to the extent possible. Our main source is HandBook of Applied Cryptography, Alfred J. Menezes, Paul C. van Oorschot and Scott A. Vanstone, which is freely available online.

Integration with the Verificatum Mix-Net

A few classes in the library are tailored for simple integration with the Verificatum Mix-Net (VMN) in which the servers jointly generate a marshalled public key that contains the group description. If this is then combined with a description of a zero knowledge proof the client would proceed with one line of code.

<script src="vts-baac-VERSION.js"></script>

const [ciphertext, digest, seed] = (new WebAPIElGamalZKPoKClient(mpk)).encrypt(label, plaintext)

Organisation

Source Code

The source code resides in src/ts and contains M4 macros defined in src/m4. You may populate a TypeScript monorepo or single-file versions of the library (or subsets of the functionality).

TypeScript Monorepo

You may use make tsmodules to populate monorepo packages with pure TypeScript code. When populated with code the monorepo contains the following packages with zero external dependencies, but with internal dependencies given by their order.

  • base contains base classes and an abstract random source.

  • arithm contains routines for arithmetic with big integers.

  • algebra contains implementations of multiplicative groups, elliptic curve groups, and abstractions for product groups.

  • crypto contains implementations of El Gamal and generalized Schnorr proofs.

  • dev contains testing and benchmarking routines.

In addition to this the following packages allow application of the library within different environments:

  • node contains a wrapper of the builtin cryptographically strong pseudo random generator of Node.

  • devnode contains a thin wrapper to run the test suite or benchmark suite from the command line. Use make [check|bench]_modules to invoke the tests or benchmarks, respectively.

  • dom contains a wrapper of the builtin cryptographically strong pseudo random generator of browsers provided by the WebAPI.

  • devdom contains a thin wrapper to run the test suite, benchmark suite, or example in a browser. Use make [check|bench]_files to invoke the tests or benchmarks, respectively.

JavaScript from Monorepo

You may use make jsmodules to compile the TypeScript modules into JavaScript modules which end up in packages/<package>/dist. A directory @verificatum is generated with symbolic links to these directories to simulate a rudimentary form of packaging that allows using the library directly from Node.

TypeScript as Single File

The precursor of this library was provided as a single JavaScript file.

Similar TypeScript files, along with files which includes code from the additional packages, can be compiled using make tsfiles. This is done using custom scripts found in tools/. These files are found in pure/src.

  • vts-ba-VERSION.ts - Base routines and large integer arithmetic.

  • vts-baa-VERSION.ts - Adds arithmetic for rings and groups.

  • vts-baac-VERSION.ts - Adds group-based cryptography.

  • vts-baacd-VERSION.ts - Adds testing and benchmarking routines.

  • vts-baac[d]-[n/d]-VERSION.ts - Adds wrapper for the cryptographically strong random generator of Node (n) or WebAPI (d) to the vts-baac[d] configuration, with (d), or without, testing and benchmarking routines.

JavaScript from Single File

The TypeScript single files may be compiled using, e.g, make vts-ba, into single JavaScript files that closely resemble the original library VJSC. The resulting files are found in pure/dist.

Minified JavaScript Variants

The JavaScript files may be minified by adding the postfix -m- to the configuration, e.g., make vts-ba-m compiles vts-ba-m-VERSION.js which is a minified version of vts-ba-VERSION.js.

We use Uglify, https://github.com/mishoo/UglifyJS. This is a relatively conservative minifier, but you should of course run the tests and benchmarks on the minified code using make check_filesm and make bench_filesm to make sure that the code that you are actually use behave as expected.

Optional WebAssembly Compiled from C

Optionally the muladd_loop() function, which is the work horse of the whole library, can be compiled from C to WebAssembly. In this case the assembly code is injected into the source of packages/arithm/src/li/func.ts to make the final library self-contained.

Run, e.g., make jsfiles WASM=true or edit the Makefile to achieve this. The source contains macros starting with M4_WASM that enables or disables, and injects different code. This gives a speedup of roughly a factor of two for large modular exponentiation, but it increases the memory requirements due to how the WebAssembler engine allocates memory.

The WASM code itself is embedded in the TypeScript source file to make it self-contained and avoid asynchronous function calls when loading.

Selecting the Included Groups

You may wish to use only multiplicative groups or only elliptic curve groups, and you may want to only include a subset of the standardized named groups. The former is achieved, e.g., using make GROUPTYPES=ModPGroup. We refer the user to the Makefile for how to choose a subset of the standardized groups.

A Word about Wordsizes

JavaScript engines uses 64 bit floating point arithmetic, semantically speaking, but internally they may use 32 bit integers if it can be inferred that computations performed are equivalent.

Thus, the implementation uses a number for each limb of a large integer as if it was a int32_t, but the actual wordsize can be configured, e.g., using make jsfiles WORDSIZE=WS, where WS is an even integer between 8 and 30 (inclusive).

This allows exhaustive testing of the most basic routines from a logical point of view and is an effective way to trigger any word boundary bugs. It also makes the code a good teaching tool for intermediate level arithmetic code.

Contributing

The source code resides in the src/ts directory. Some TypeScript code is generated using M4 and possibly custom scripts. If you open an issue or make a pull request, then please make sure that you do it for the source and not the generated TypeScript code. The generated TypeDoc pages point to the actual source to help you do this.

Remarks

Generally, the web is a moving target and many tools are in development. This either requires workarounds, or workarounds are more stable and less error prone than using the most recent features announced. Incompatibilities between Node and browsers means that we have implemented some functionality that (strictly speaking) is already available, but this improves portability and robustness.

TypeDoc is incomplete and does not handle dependencies correctly. Thus, we generate a temporary source tree that it can process correctly to generate the documentation.

There no standard way of bundling JavaScript into a self-contained library, and the existing solutions are evolving. Thus, we leave it to the user to choose how to bundle the code using the module system they prefer, but we provide a standard monorepo structure as a starting point.

Our single-file versions of natural subsets of the library are generated using custom scripts which rely on the consistency of our software. You may edit the Makefile to pick an even more precise subset, but please be careful.

We consider the single-file versions to be the most stable versions of the software, since they depend less on volatile decisions made by other developers and allow standalone testing.

Future

If you have a better solution to any of the hacks used, then feel free to reach out to us, but please do not suggest yet another unstable package for something that can be done with a one-liner shell script.

Our hope is that TypeScript will introduce stricter types similar to those available in AssemblyScript. The source is written with this in mind, so it is typed "optimistically" more stringently than necessary, e.g., using int32_t. These "types" should be considered to be no more than helpful comments at the moment, since no type checking actually occurs that differentiates integer types or even numbers from integers. Branding, interfaces, or union types do not actually give integer types.

History

VTS is a manual transpilation and improvement of the Verificatum JavaScript Cryptography library (VJSC). The main refactoring done is isolating the integer arithmetic routines to allow independent use and moving implementations of groups and rings from arithm to algebra.

Apart from this the transpilation is faithful, but the code is carefully organized to minimize dependencies and allow subsets to be used separately. No bugs have been discovered, but the software has been made more robust against incorrect usage. Modern language constructs introduced after VJSC was written are used consistently. The documentation has been improved and examples have been added to simplify adoption and deployment.
