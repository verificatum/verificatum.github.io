@verificatum/bundle

Self-contained cryptographic library for use in electronic voting clients. Complete documentation with explicit references to the literature on basic arithmetic algorithms makes it good teaching material.

Installation

Installation instructions are given for Ubuntu. Adapt as needed.

# Building tools.
sudo apt install make m4

# Most recent version of node.
curl -fsSL https://deb.nodesource.com/setup_current.x | sudo -E bash -
sudo apt install -y nodejs

# TypeScript globally.
sudo npm install -g typescript

# Install NPM package dependencies.
npm i

# To enable WASM code the following is needed.
sudo apt install clang
sudo update-alternatives --install /usr/bin/wasm-ld wasm-ld /usr/bin/wasm-ld-12 100

# To run the webserver that serves the test and benchmark suites you
# may need to install Python 3.
sudo apt install python3

Usage

The simplest way to use the library (with or without WASM code) is to use one of the single-file JavaScript files compiled using make jsfiles which end up in pure/dist, but users may also choose to use a bundling tool that understands monorepos.

TypeDoc Documentation

The TypeDoc documentation can be built using make doc and end up in the subdirectory doc, but it is also available here. We have modified the output to link directly to the relevant lines in the actual source code at GitHub.

Example

After installing the build tools above you can build the library and start a webserver at localhost where you can try it directly. Simply use make webserver to compile the code and start the webserver. The example, benchmark suite, and test suite are then available at:

  • Example Client
  • Benchmark Suite
  • Test Suite

The source code of the example is heavily documented to allow simple cut and paste and usage in your own client, which hopefully has a more impressive design.

Organisation

Source Code

The source code resides in src/ts and contains M4 macros defined in src/m4. You may populate a TypeScript monorepo or single-file versions of the library (or subsets of the functionality).

TypeScript Monorepo

You may use make tsmodules to populate monorepo packages with pure TypeScript code. When populated with code the monorepo contains the following packages with zero external dependencies, but with internal dependencies given by their order.

  • eio contains packaging routines and base class for a random source.

  • arithm contains routines for arithmetic with big integers.

  • algebra contains implementations of multiplicative groups, elliptic curve groups, and abstractions for product groups.

  • crypto contains implementations of El Gamal and generalized Schnorr proofs.

  • dev contains testing and benchmarking routines.

In addition to this the following packages allow application of the library within different environments:

  • node contains a wrapper of the builtin cryptographically strong pseudo random generator of Node.

  • devnode contains a thin wrapper to run the test suite or benchmark suite from the command line. Use make check_modules to invoke it.

  • dom contains a wrapper of the builtin cryptographically strong pseudo random generator of browsers provided by the WebAPI.

  • devdom contains a thin wrapper to run the test suite, benchmark suite, or example in a browser.

TypeScript as Single File

The precursor of this library was provided as a single JavaScript file.

Similar TypeScript files, along with files which includes code from the additional packages, can be compiled using make tsfiles. This is done using custom scripts found in tools/. The test suite can be run for this file using make check_single.

JavaScript from Monorepo

You may use make jsmodules to compile the TypeScript modules into JavaScript modules which end up in packages/<package>/dist. A directory @verificatum is generated with symbolic links to these directories to simulate a rudimentary form of packaging that allows using the library directly from node.

JavaScript from Single File

The TypeScript single files may be compiled using make jsfiles into single JavaScript files that closely resemble the original library VJSC.

Optional WebAssembly

Optionally the muladd_loop() function which is the work horse of the whole library can be compiled from C to WebAssembly, in which case the assembly code is injected into the source of packages/src/arithm/li/func.ts. Edit Makefile to achieve this. Search for WASM_M4 and comment/uncomment two lines as explained. The source contains macros starting with M4_WASM that enables or disables, and injects different code.

This gives a speedup of roughly a factor of two for large modular exponentiation, but it increases the memory requirements due to how the WebAssembler engine allocates memory.

The WASM code itself is embedded in the TypeScript source file to make it self-contained and avoid asynchronous functions, but it requires calling verificatm.arithm.li.initialize(), since we wish to allow the use of code diversification, i.e., using different instantiations of this code.

Contributing

The source code resides in the src/ts directory. The TypeScript code is generated using M4 and possibly custom scripts. If you open an issue or make a pull request, then please make sure that you do it for the source and not the generated TypeScript code.

Remarks

Generally, the web is a moving target and many tools are in development. This either requires workarounds, or workarounds are more stable and less error prone than using the most recent features announced. Incompatibilities between Node and browsers means that we have implemented some functionality that strictly speaking is already available, but this improves portability and robustness.

TypeDoc is incomplete and does not handle dependencies correctly. Thus, we generate a temporary source tree that it can process correctly to generate the documentation.

There no standard way of bundling JavaScript into a self-contained library, and the existing solutions are evolving. Thus, we leave it to the user to choose how to bundle the code using the module system they prefer, but we provide a standard monorepo structure as a starting point.

Our single-file versions of natural subsets of the library are generated using custom scripts which rely on the consistency of the software. You may edit the Makefile to pick an even more precise subset, but please be careful.

We consider the single-file versions to be the most stable versions of the software, since they depend less on volatile decisions made by other developers and allow standalone testing.

History

VTSC is a manual transpilation and improvement of the Verificatum JavaScript Cryptography library (VJSC). The main refactoring done is moving implementations of groups and rings from arithm to algebra. Apart from this the transpilation is faithful, but the code is organized to minimize dependencies. No bugs have been discovered, but the software has been made more robust against incorrect usage. Modern language constructs introduced after VJSC was written are used consistently. The documentation has been improved and an example has been added to simplify deployment.

Future

If you have a better solution to any of the hacks used, then feel free to reach out to us, but please do not suggest yet another unstable package for something that can be done with a one-liner shell script.

Our hope is that TypeScript will introduce stricter types similar to those available in AssemblyScript. The source is written with this in mind, so it is typed "optimistically" more stringently than necessary. These "types" should be considered as helpful comments at the moment, since we can not verify their usage.
